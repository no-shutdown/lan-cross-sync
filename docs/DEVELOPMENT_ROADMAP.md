# LAN Cross Sync 开发交接路线图

本文档是从当前仓库继续开发的入口。设计原则和用户需求以
[`superpowers/specs/2026-07-23-lan-cross-device-sync-design.md`](superpowers/specs/2026-07-23-lan-cross-device-sync-design.md)
为准；本文档负责说明开发顺序、交付边界和验收方式。

## 当前基线

当前分支：`master`

当前基线提交：`db7939d docs: add discovery receive implementation plan`

已经完成：

- Tauri v2 + React + TypeScript + Rust 项目骨架。
- 本机设备 ID 和设置持久化。
- 版本化 UDP 局域网发现广播与接收。
- 发现设备列表、已配对设备列表和连接状态注册表。
- 6 位临时配对码状态机及过期处理。
- Tauri 控制命令、基础控制面板、托盘驻留和开机启动开关。
- 每设备接收剪贴板开关、默认文件目标数据模型。

当前仍不能完成：

- 两台真实电脑之间的配对握手和可信设备交换。
- 剪贴板文本/图片同步。
- 文件、文件夹传输和接收端保存对话框。
- Mac/Windows 跨机器验收、安装包和权限引导。

## 迭代 2：配对与可信设备交换

目标：把当前“本地生成配对码”变成真实的双机流程。设备 A 展示一次性代码，设备 B 选择设备 A 并输入代码；验证成功后双方都保存对方的 `PairedPeer`。

建议任务：

- 补齐配对请求、响应、目标设备 ID、代码和协议版本字段，并更新 wire-format 测试。
- 在现有 UDP 接收循环中分发配对消息，严格过滤目标设备、协议版本和自身设备。
- 记录请求来源地址或设计明确的广播响应机制，避免依赖不可达的设备地址。
- 增加 `start_pairing`、`request_pairing`、`accept/reject` 或等价的 Tauri 命令。
- 在发现设备卡片上增加输入配对码和错误状态；成功后刷新已配对列表。
- 配对成功后持久化双方设备信息，保留未来公钥/密钥字段的位置。

完成定义：

- 两个本地实例可完成成功、错误代码、过期代码、取消和重复配对。
- 重启应用后配对记录仍存在，未配对设备不能进入后续传输。
- Rust 单元测试覆盖状态转换和协议拒绝；至少一次双实例本机集成测试通过。

## 迭代 3：基础传输层

目标：建立配对设备之间可复用的可靠控制通道。发现仍负责“找到设备”，传输层负责连接、消息路由和生命周期。

建议任务：

- 新建 `transport` 模块，定义连接句柄、消息 framing、协议版本检查和最大消息大小。
- 采用控制消息与文件字节分离的结构；先实现控制消息，不在内存中缓存整个文件。
- 实现连接、心跳、超时、断开、网络恢复后的重连和状态更新。
- 对未经配对的设备拒绝业务消息；所有日志不得记录剪贴板正文、图片字节或文件内容。
- 将发送/接收错误映射成 UI 可理解的错误码。

完成定义：

- 配对设备可看到 `connected`，关闭另一实例后变为离线，重新启动后自动恢复。
- 大量无效消息不会导致崩溃或无限内存增长。
- 连接和协议核心逻辑有单元测试，至少有双实例连接测试。

## 迭代 4：文本剪贴板同步

目标：实现“直接复制，另一台直接粘贴”的核心体验。

建议任务：

- 新建 `clipboard` 模块，监听本地文本变化并生成事件 ID、序号、时间戳、hash 和来源设备 ID。
- 只向已配对且在线、并开启 `receive_clipboard` 的设备发送。
- 接收端写入系统剪贴板；记录最近写入的远端事件，避免回写形成循环。
- 处理快速连续复制、重复内容、空内容、系统剪贴板读写失败。
- 增加后台任务启动/停止和 Tauri 事件通知，UI 只展示状态与错误。

完成定义：

- Mac 到 Windows、Windows 到 Mac 的文本复制粘贴均通过。
- 关闭某设备的接收开关后不会写入该设备剪贴板。
- 环路去重、事件顺序和失败重试有测试。

## 迭代 5：图片剪贴板同步

目标：在文本同步稳定后支持复制图片。

建议任务：

- 统一文本和图片的剪贴板事件模型，图片使用 MIME、尺寸、hash 和受控 payload。
- 设定明确的 MVP 图片大小上限，超过上限时提示改用文件传输。
- 对图片进行必要的格式转换和平台适配，不记录图片内容。
- 增加图片循环去重、损坏 payload、超过限制和写入失败测试。

完成定义：

- 常见截图可在 Mac/Windows 双向复制粘贴。
- 超限图片不阻塞连接，也不会被部分写入。

## 迭代 6：文件和文件夹传输

目标：用户把文件或文件夹拖到固定区域，选择目标设备；接收端选择保存位置后开始传输。

建议任务：

- 将前端 drop zone 接入 Tauri 文件拖放事件，扫描文件/文件夹并生成相对路径树。
- 增加传输任务、元数据、分块流、进度、取消、整任务重试和失败状态。
- 接收端弹出保存位置确认；拒绝、路径不可用、重名和目录创建失败都要有明确结果。
- 控制消息与文件数据分离，限制并发和内存占用。
- 传输完成后提供打开目标文件夹的操作。

完成定义：

- 单文件、多文件、嵌套文件夹都能保留相对目录结构。
- 大文件传输过程中 UI 有稳定进度，取消后双方状态一致。
- 不实现断点续传；失败时明确提示重新传输整个任务。

## 迭代 7：产品化和跨平台验收

建议任务：

- 完成 Mac 菜单栏、Windows 托盘、关闭窗口隐藏、开机启动的真实验收。
- 增加 Mac 本地网络/通知/剪贴板权限说明，以及 Windows 防火墙放行说明。
- 配置生产 CSP，检查 Tauri capabilities 最小权限和文件系统范围。
- 生成 Mac 和 Windows 安装包，确认升级、卸载、设置保留策略。
- 建立真实双机验收表：发现、配对、重启重连、文本、图片、文件、文件夹、睡眠唤醒。
- 清理 deferred API 的死代码警告，保留必要的未来扩展接口。

当前已知技术债：`tauri.conf.json` 的 CSP 仍为开发期配置；Rust 中有部分为后续传输/配对流程预留的未使用 API。它们不阻塞当前基础功能，但在发布前必须处理。

## 迭代 8：安全与增强

按优先级考虑：

- 配对期间密钥交换和可信设备认证。
- 控制消息与文件传输的端到端加密。
- 断点续传、带宽限制和更细的冲突处理。
- 剪贴板历史、浮动拖放窗口、系统分享菜单。

安全功能完成前，产品定位应明确为“可信局域网内使用”，不要对外宣称可抵御局域网攻击。

## 接手开发约定

- 每个迭代先新增一份 `docs/superpowers/plans/YYYY-MM-DD-<topic>.md`，写清任务和验收标准。
- 每个任务先写失败测试，再实现，再运行格式化、Rust 测试和前端构建。
- 协议变更必须更新 wire-format/golden JSON 测试，并说明兼容策略。
- 业务逻辑放 Rust 模块；React 只负责展示、输入和 Tauri 命令调用。
- 每个迭代保持可编译、可回滚，并使用单一主题的 Conventional Commit。
- 交接前执行：

```powershell
git status --short
C:\Users\13565\.cargo\bin\cargo.exe test --manifest-path src-tauri\Cargo.toml
pnpm build
```

- 需要真实网络验证时，用两台设备分别运行开发包，并记录系统、防火墙、局域网和测试结果；不要只用单元测试替代双机验收。

## 推荐下一步

从迭代 2 开始，先完成真实配对闭环，再进入传输层。不要在配对记录尚未真正可信之前实现剪贴板和文件业务，因为这会让权限边界、目标设备选择和断线行为重复返工。
