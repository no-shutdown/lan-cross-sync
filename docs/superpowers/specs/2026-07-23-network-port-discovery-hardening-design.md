# 网络端口与设备发现加固设计

## 背景

当前应用把设备声明的端口同时用于 UDP 发现和 TCP 传输。广播发送端使用临时 UDP 端口，注册表却保存广播报文的源端口，因此配对请求和 TCP 连接可能访问错误的端点。启动时各网络任务也独立绑定，端口冲突只写入日志，用户无法在界面判断网络服务是否可用。

## 目标

- 固定 UDP 发现端口为 `45731`，所有设备用同一个端口接收发现和配对报文。
- 将 TCP 传输端口与 UDP 发现端口分离。TCP 优先使用设置中的端口，被占用时自动申请空闲端口。
- 只有 UDP 接收端点和 TCP 监听端点都成功绑定后才开始广播本机设备。
- 将实际 TCP 端口写回本地设置，并在发现报文中继续通过 `DeviceInfo.port` 广告该端口。
- 注册表分别保存 UDP 发现端点和 TCP 传输端点，配对走 UDP，数据传输走 TCP。
- 将绑定失败、端口回退和广播状态展示在界面中。
- 通过 TCP `Hello/HelloAck` 完成应用协议和已配对身份校验；发现报文只作为候选设备信息。

## 非目标

本次不引入端到端加密，也不解决设备 UUID 自报导致的恶意身份伪造。现有协议仍假设运行在可信局域网内。

## 启动流程

1. 读取本地设置，使用 `local_device.port` 作为 TCP 首选端口。
2. 尝试绑定 TCP 首选端口；失败时绑定 `0`，取得操作系统分配的实际端口，并保存到设置。
3. 绑定固定 UDP 发现端口 `45731`。
4. 两个端点都成功时启动 UDP 接收、TCP 监听和 UDP 广播任务；否则不广播本机设备，应用仍启动并展示错误状态。

## 端点模型

- `discovery_endpoint`: 对端 IP + `45731`，用于发现、配对请求、配对响应和确认。
- `transport_endpoint`: 对端 IP + 发现报文中声明的 `DeviceInfo.port`，用于 TCP 连接和应用层握手。
- UDP 广播发送端可继续使用临时源端口，因为它不再被当作对端端点保存。

## 错误与可观测性

运行时状态包含 UDP 发现端口、TCP 实际端口、两个端点是否就绪、是否正在广播以及可本地化的状态代码。原始绑定错误写入日志，界面只显示稳定的状态文案，避免把本机路径或系统细节暴露给用户。

## 验证

- 注册表测试验证发现端点固定为 `45731`，传输端点使用设备声明的 TCP 端口。
- 传输绑定测试占用首选端口，验证自动回退到不同的空闲端口。
- Rust 单元测试、格式化、前端构建和 Tauri release 构建均通过。
- 手工验证：启动两个实例，确认 UDP 发现和配对仍可用；占用默认 TCP 端口时确认界面显示回退状态，且广播报文携带实际 TCP 端口。
